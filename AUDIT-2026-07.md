# cue.mith.studio — Comprehensive Product, Engineering & Market Audit

**Date:** July 2026 · **Scope:** full codebase audit, UX audit, competitive research (live-show + dance markets, ~140 web sources), strategic assessment and forward plan.
**Method:** read-only code audit against the live system (confirmed against the running DB), 13 parallel research passes, synthesized here. This document is the working reference for future development phases.

---

## 1. Executive summary

**The product idea is validated and the position is genuinely rare. The execution is strong at the line level and reckless at the operational level. The biggest risk is not a competitor — it is that the tool is being built faster than it is being used.**

Five sentences that capture the whole audit:

1. **The market whitespace is real, on both fronts.** Nothing on the market combines web-based + realtime-collaborative + waveform/beat-grid-native + multi-department cues + console/DAW export. On the dance side, no tool anywhere combines count/waveform annotation with formation animation — the two ecosystems are completely disjoint today.
2. **Engineering craft is high** — the realtime visibility handling, versioned media pipeline, and capability RBAC are better plumbing than several funded competitors — **but the flagship "rollback-safe versioning" claim is partly false** (rollback-restore has been broken since Phase 2b) and one of the two shipped Phase-11a trust fixes is dead code. Both went unnoticed because there is effectively **no test coverage** (1 test file in 17k LOC).
3. **The operational posture is existential-risk-grade:** the entire codebase is **untracked in git** (exists only on one VPS disk), there are **no database backups of any kind**, and S3 objects orphan forever on delete. One disk failure destroys the product.
4. **Usage is near zero** — 6 users, 14 projects, 41 cues — while **five fully-built features have zero adoption** (playlists, share links, crew assignments, dept-scoped roles, markers). Feature velocity has decoupled from user acquisition.
5. **The UX is a power tool with no product clothes:** ~45 controls on one screen, sign-up hidden inside "Forgot password", developer vocabulary throughout ("pt/rgn", "analyzer down", "Rollback +cues"), and the dance persona is a second-class tenant in a DJ-shaped interface.

**Verdict: right path strategically, wrong path procedurally.** The recommended pivot is not in what the product is, but in how the next six months are spent: one week of stop-the-bleeding ops/bug work, then a deliberate shift from ~100% building to ~50% building / 50% getting real crews and schools onto it, with the roadmap re-cut around what those users hit first.

### 2.2 Confirmed bugs (verified against live schema/DB)

1. **Rollback-restore is broken and always has been (since Phase 2b).** `server/services/versions.js:102-127` re-inserts snapshot cues **without `lane_id`**, which is NOT NULL — every `restoreCues: true` rollback 500s (transaction rolls back, no corruption, but the feature has never worked post-Phase-2b). Even fixed, the field map silently drops status, OSC, MIDI, automation and lane — snapshot restore would lose production programming.
2. **Phase 11a mail-failure logging is dead code.** `mail.js:46-50` calls `logActivity` with no `projectId` (NOT NULL) and an `action` value not in the native enum — the insert always fails and is swallowed. Prod mail failures are exactly as invisible as before.
3. **Team members are denied realtime.** REST grants access via team membership, but the socket `join-project` handler (`socket/index.js:48-53`) only checks `project_members` — team-based members silently get no live collaboration.
4. **Cross-project authorization gap (IDOR class).** `loadMembership` authorizes the *URL's* project, but version endpoints (`tracks.js:251, 400, 444, 471` — including presigned media URLs) and **every PATCH/DELETE-by-id** in cues/markers/showElements/formationPlacements never verify the target row belongs to that project. A member of project A with `edit_others_cues` can mutate cues in project B given UUIDs. The `trackInProject()` helper already exists in all four files — it's just only called on GET/POST.
5. **Beat-grid drift.** Reprocess only backfills NULL beats (`worker.js:130-138`); a BPM change (the exact analyzer-retry flow shipped in 11a) strands every existing cue's `start_beat` against the new grid. `lib/beats.js` is a constant-BPM linear model; the stored rekordbox variable beatgrids are never used.
